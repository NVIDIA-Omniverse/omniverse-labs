# usd-developer-skillgraph

> **Part of [`nanousd-labs`](..).** nanousd-labs is an experimental fleet — skills and prompts that generate USD implementations and stacks from the [USD Core Specification](https://github.com/aousd/specifications-public/tree/main/core). This component is the **heart: the methodology substrate — skills, contracts, and goldens — that an agent walks to generate USD library code from the spec.** The [fleet README](..) carries the umbrella story.

`usd-developer-skillgraph` is a skill graph — skills, contracts, and goldens — pinned to the [USD Core Specification](https://github.com/aousd/specifications-public/tree/main/core). An agent walks the graph to generate USD library code: the spec is the durable input; the generated implementation is an elastic, regenerable output per target language or platform.

Contracts define what correct looks like: implementation-neutral interfaces and performance/storage invariants that independently-generated implementations agree on, without either copying the other. The skill graph is the heart of the fleet; the generated backends it produces conform to the same C ABI as [`nanousd`](../nanousd), the fleet's flagship implementation.

## What it generates today

The graph generates working **single-layer USD parsers** — both the text (`.usda`) and binary (`.usdc`) formats, in Python and C++ — that open a resolved local layer resource and parse it into the canonical document-model dump.

Beyond the parsers, the graph covers, as contracts and goldens that bound what each node generates:

- **Stage population** — a namespace-source boundary spanning single-layer, sublayer, reference, payload, variant, schema, and instancing providers.
- **Composition** — direct providers for the LIVERPS arcs plus a central arbitrator. These consume already-supplied sources; recursive arc discovery and asset resolution and opening are not yet covered.
- **Value resolution** — a resolver over already-composed opinion stacks: attribute defaults, value blocks, time samples, splines, value clips, schema fallbacks, and metadata and listOp resolution.
- **Schema** — a schema-aware namespace-source wrapper: typed and applied schema property injection, base inheritance, `fallbackPrimTypes` substitution, and schema-defined fallbacks for value resolution.
- **USDZ** — a single-layer package scope that opens one layer from a local USDZ package, selecting the package default layer or an explicit entry and dispatching the `.usda`/`.usdc` bytes to the existing handlers. Broader Crate coverage and USDC writing are future work.
- **nanousd backend** — a read-only C-API backend contract that plugs into `nanousd`'s compliance test framework via the same C ABI. Write/create paths and full recursive runtime composition — wiring the composition, schema, and value-resolution nodes into one generated runtime — are deferred.

This is the skill graph's own generative coverage — distinct from the flagship [`nanousd`](../nanousd) implementation, which was generated by engineer-guided agents prompting directly against the spec (not from this graph) and is further along. Contracts precede generated code — they define what correct looks like before an agent generates against them. `graph/scopes/` is the source of truth for what's active.

Generated implementations belong under `generated/`, are disposable, and are `.gitignored` — **a fresh clone has none, by design.** You produce a target by walking the graph with an agent (start with the [tutorial](docs/tutorial/README.md)); the pinned spec excerpts it reads are likewise materialized locally under `spec/`, not checked in. Durable changes go to skills, contracts, goldens, or the graph.

**New here?** Walk the **[10-minute tutorial](docs/tutorial/README.md)** — tour a handful of skills and contracts, then run the core loop yourself: watch a case fail, reach for the contract that owns it, and watch it pass. It ends on the shape of how to make your own contribution.

## Layout

```text
graph/          skill graph, executable manifest, and target scopes
skills/         skill instructions per graph node
contracts/      implementation-neutral output contracts (handles, capabilities, performance, spec-coverage)
goldens/        manually authored conformance cases
harness/        scoring, benchmark, and regeneration drivers
benchmarks/     benchmark target definitions and fixtures
generated/      disposable generated targets (gitignored)
reports/        run reports and ambiguity notes
spec/           materialized pinned spec excerpts
docs/           contract stack documentation per scope
```

## Running the graph

> Commands use `python3` (Linux/macOS); on Windows use the `py` launcher instead.

Inspect the generation order and dependency closure for the primary target:

```bash
python3 harness/regen_graph.py --scope usda-single-layer --target python
```

Record already-generated artifacts and run full subgraph validation:

```bash
python3 harness/regen_graph.py --scope usda-single-layer --target python --record-existing --validate
```

The same graph targets C++:

```bash
python3 harness/regen_graph.py --scope usda-single-layer --target cpp --validate
```

The intended workflow: ask for a node or scope → expand its transitive dependency closure in topological order → skip nodes whose artifacts and dependency fingerprints still match → regenerate missing or stale nodes → run the scope's full golden and benchmark suite.

See `REGEN_TASK.md` for the operator runbook and `docs/` for per-scope contract stack documentation.

## Scoring

A generated target exposes a dump command that accepts a USDA filepath as its final argument and writes JSON to stdout:

```bash
python3 harness/score.py \
  --goldens goldens/integration/usda-single-layer/basic.json \
  --dump-cmd "{python} generated/python/dump_layer.py"
```

`{python}` is a portable placeholder for the active interpreter. The scorer can also compare precomputed output files via `--actual-dir reports/actual`.

## Spec pin

The graph is grounded in [USD Core Specification v1.0.1](https://github.com/aousd/specifications-public/tree/main/core). The exact commit pin and materialization instructions are in `spec/README.md`. Use `materialize-spec.py` (or the `.sh`/`.ps1` wrappers) to copy the relevant spec sections into `spec/pinned`. The materialized spec text is a local working artifact, not the primary project source.

## Boundary rule

The graph declares what each node depends on; skill files also contain **boundary guards** that prevent an agent from recreating base infrastructure locally. For example, `usda-spec-parser` uses the path and document-model contracts rather than inventing a second path representation.

## Contributing

This component is published as part of an experiment under Omniverse Labs and isn't set up to take inbound pull requests. The invitation is to the **method**: take the skills, contracts, and goldens here as the starting point for generating your own implementations, and file spec-ambiguity reports upstream on the [AOUSD specifications repo](https://github.com/aousd/specifications-public).
