> current-state picture of the nanoUSD experiment: what works today, where we're
> taking it, and the problems we're working through to get there. It states
> current state — not change history — and is refreshed roughly monthly, so
> expect the picture to move.

## Where we are

Everything in the fleet is generated by agents. It's several repositories, but the
work in them splits into two very different stages — and what separates them is *how*
the generation is driven.

**1. The `nanousd` library and the stack built on it — usable today.** A small,
portable C/C++ library that reads, combines, queries, and writes USD scene files.
Clone it, build it, call it from C or Python. It was generated by an engineer
directly prompting an agent against the spec — not from the skill graph's library of
skills — and it's the most complete part: it handles USD's full layering/composition
rules and writing, and it passes a compliance test suite cross-checked against OpenUSD. The renderers, the
viewer, and the Python bindings are all part of this — they sit on top of the
library. (That's most of the repos here.)

**2. The skill graph — the experiment, and still early.** This is the experiment at
the heart of the project: instead of generating the code from one-off prompts (the
way `nanousd` was built), you distill the standard into a reusable library of skills
the agent draws on to generate implementations. That happens in two
steps, and the difference between them is the key to reading everything else:

1. **Define "correct"** — write down, straight from the standard, exactly what a
   correct implementation of a feature has to do: the tests and rules it must pass.
2. **Generate** — have an agent write code that actually passes those tests.

A feature can be at step 1 (we've pinned down what correct means) without being at
step 2 (working generated code that passes). That gap — *defined* vs.
*generated-and-passing* — is the single most useful thing to understand here.

So, concretely:

- ✅ **Generated and working today:** reading a single USD layer, end to end — both
  the text (`.usda`) and binary (`.usdc`) formats, generated from the standard against
  the same compliance suites, each targeting Python and C++.
- 🟡 **Defined, generation still in progress:** we've written down "what correct
  looks like" for a lot more — combining layers, resolving values, variants, schemas,
  instancing, packaged `.usdz` files — but agent-generated code that passes all of it,
  end to end, isn't there yet.
- 🔴 **Not yet:** authoring, and — the real frontier — stitching many features
  together into one coherent implementation: generating the *whole* library from
  skills. Any one feature generates well on its own; making them work together is
  the hard part.

Where to look: each feature's "definition of correct" is a contract plus goldens in
[`usd-developer-skillgraph`](./usd-developer-skillgraph),
with a matching skill that drives the generation — browse its `contracts/`, `goldens/`,
and `skills/` to see them.

When the skill-graph-generated backend is run through the *same* compliance tests as
`nanousd` (and against OpenUSD), disagreements between the implementations surface as
test signals — and that cross-checking is what makes "generate it from the standard"
credible.

So today the library and the generator are two tracks that meet at the same C
interface — not one producing the other.

## Where we want to be

- The agent regenerates the library — and other implementations and stacks — directly
  from the standard, each passing the same standard-derived tests. *Specs are
  durable. Agents are elastic.*
- The code becomes a regenerable output you re-make for your own constraints; people
  own the judgment calls — performance, tradeoffs, design.
- The two tracks merge: the prompt-generated library becomes something the method can
  regenerate from skills, so "the library" and "the generator" stop being separate stories.

## The challenges we're working through

- **We're still inventing the method itself.** Writing skills as standalone text
  rules turned out not to be enough — and can even steer an agent wrong. The
  knowledge that actually sticks is ending up in two places: the precise "definition
  of correct" for the library, and structured, guided setups for the renderers. Same
  lesson reached from two sides.
- **Cohesion is the frontier.** Any one feature generates well on its own; the hard,
  still-unsolved part is making many features work together as one coherent whole.
- **What "skills" become is open.** The durable asset is the human-written layer that
  steers the agent; its eventual shape — and even what to call it — is still forming.
