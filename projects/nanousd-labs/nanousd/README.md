# nanousd

> **Part of [`nanousd-labs`](..)** — an experimental fleet that generates USD implementations and stacks from the [USD Core Specification](https://github.com/aousd/specifications-public/tree/main/core). This component is the **flagship — a portable C-ABI implementation of the Core Spec**. The [fleet README](..) has the full picture — the stack, how the repos fit together, and the skillgraph that drives it.

`nanousd` is a lightweight, portable implementation of the [USD Core Specification](https://github.com/aousd/specifications-public/tree/main/core). The core is C++17; the public surface is a stable-in-shape C11 API (`nanousdapi`) that binds cleanly from any language. It is a **data layer** — parse, compose, query, write — deliberately disaggregated from the rendering, imaging, and GUI layers, so it stays small enough to embed where the full OpenUSD stack can't go.

It is **not** a replacement for OpenUSD — your OpenUSD stack keeps working untouched; use what fits, file what doesn't. `nanousd` is a differently-shaped USD: a small, auditable data layer scoped to the Core Spec's runtime data model.

## How it's built

`nanousd` was **generated by engineer-guided agents prompting directly against the** [USD Core Spec](https://github.com/aousd/specifications-public/tree/main/core). Building it this way is one experiment; the more aspirational, frontier one is the skill graph ([`usd-developer-skillgraph`](../usd-developer-skillgraph)) — a *method* that **generates** USD implementations from the standard, shifting source code further left. Today it generates only a slice; how far that can go is the open question. The [fleet README](..) shows what works versus what doesn't yet.

The [`skills/`](skills/) folder here is **not** the skill graph — it's a single repo-local scene-authoring tool (`sketch-stage`), and you don't need it to use the library. See [`skills/README.md`](skills/README.md).

## Dual-backend design

The C API (`nanousdapi`) is a thin dispatch layer that forwards every call through a backend. Two backends exist:

- the built-in **nanousd** core (default — no OpenUSD dependency), and
- an optional **OpenUSD** backend that serves the same C API from a prebuilt OpenUSD (pxr) distribution.

Same API, two implementations — so the compliance suite runs against both, and any behavior divergence is a CI signal rather than a surprise. The [`usd-developer-skillgraph`](../usd-developer-skillgraph) can also generate additional backends that conform to the same C ABI and are exercised through the same compliance test framework.

## Scope and status

> **Experimental.** Pre-1.0, no stability or support guarantees. The C API is stable *in shape* (opaque handles, a narrow function list), but individual functions may change before 1.0 — pin to a commit if you need reproducibility.

`nanousd` implements the [USD Core Specification](https://github.com/aousd/specifications-public/tree/main/core) runtime data model:

- **File formats:** USDA (text), USDC (binary crate), USDZ (package) — read and write.
- **Composition:** full LIVERPS — all seven arcs (Local, Inherits, Variants, Relocates, References, Payloads, Specializes), strength ordering, chained/diamond/nested cases, implied inherits and specializes, internal references. Scene-graph instancing.
- **Value resolution:** time samples, splines, value clips, and listOp composition.
- **Schemas:** `apiSchemas`/`HasAPI` handling and typed-schema field resolution for the core set.

Out of scope: Hydra, imaging, and GPU rendering pipelines — nanoUSD is a data layer; the fleet's renderer repos sit on top of the C ABI. The test suites are the source of truth for current coverage.

## Requirements

- CMake 3.16+
- C++17 compiler
- C11 compiler (the public C API is C11)
- (Optional) A prebuilt [OpenUSD](https://github.com/PixarAnimationStudios/OpenUSD) distribution for the OpenUSD backend

## Building

### nanousd (default backend)

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --parallel
```

This produces:

| Target | Description |
|---|---|
| `nanousd` | Shared library — C++17 core (parsing, writing, composition, stage) plus the built-in backend |
| `nanousdapi` | C API dispatch layer — thin library that loads a backend and forwards all public C API calls through it |

### OpenUSD backend

The OpenUSD backend delegates to a prebuilt OpenUSD (pxr) distribution instead of using nanousd's built-in parser and composer. To build it, you must have a prebuilt OpenUSD distribution available at `backends/OpenUSD` (or another path of your choosing).

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Release \
    -DNANOUSD_BUILD_OPENUSD_BACKEND=ON \
    -DOPENUSD_ROOT=backends/OpenUSD
cmake --build build --parallel
```

The `OPENUSD_ROOT` must point to an OpenUSD installation containing `include/pxr/pxr.h` and the monolithic shared library (`libusd_ms.so` on Linux, `usd_ms.lib` on Windows).

To use the OpenUSD backend at runtime, set the `NANOUSD_BACKEND` environment variable:

```bash
NANOUSD_BACKEND=./libopenusd_backend.so ./my_app
```

### Build options

| Option | Default | Description |
|---|---|---|
| `NANOUSD_BUILD_TESTS` | `ON` | Build both test suites |
| `NANOUSD_SHARED_API` | `OFF` | Build `nanousdapi` as a shared library instead of static |
| `NANOUSD_BUILD_OPENUSD_BACKEND` | `OFF` | Build the OpenUSD backend (requires `OPENUSD_ROOT`) |
| `NANOUSD_ENABLE_SANITIZERS` | `OFF` | Enable compiler sanitizers for dev/CI test builds only |
| `NANOUSD_SANITIZERS` | `address;undefined` | Semicolon-separated sanitizer list. Supported: `address`, `undefined`, `thread` |
| `OPENUSD_ROOT` | (empty) | Path to a prebuilt OpenUSD installation |

## Using the C API

`nanousd` exposes a stable C ABI in [`include/nanousd/nanousdapi.h`](include/nanousd/nanousdapi.h). The minimal arc is **open → walk prims → close**: `nanousd_open(path)` returns a `NanousdStage` (check `nanousd_isvalid`); reach prims with `nanousd_nprims`/`nanousd_prim` (or `nanousd_primpath(stage, "/Root")`) and inspect them with `nanousd_path`/`nanousd_name`/`nanousd_typename`; release each prim handle with `nanousd_freeprim` and finish with `nanousd_close`.

For a complete, runnable version of exactly that arc, see the **`test_prim_traversal`** function in [`tests/compliance/compliance_test.c`](tests/compliance/compliance_test.c) — a short, pure-C11 open→iterate→close example that CI builds and runs, so it can't drift from the headers. (The surrounding file is the full compliance harness; `test_prim_traversal` is the part to read first.)

To build the library, then compile, link, and run your own program against it:

```bash
# Produces the static C API lib build/libnanousdapi.a and the default
# backend build/<Config>/libnanousd.so:
cmake -B build -DCMAKE_BUILD_TYPE=Release && cmake --build build --parallel

# nanousdapi is a C++ static library, so a C program must also link the C++
# runtime (-lstdc++) and, on Linux, the dynamic loader (-ldl):
cc your_program.c -Iinclude -Lbuild -lnanousdapi -lstdc++ -ldl -o app

# At runtime nanousdapi loads its backend; point NANOUSD_BACKEND at the
# default backend built above (it lives in build/<Config>/):
NANOUSD_BACKEND=build/Release/libnanousd.so ./app scene.usda
```

> **Runtime backend discovery.** `nanousdapi` dispatches to a backend it loads at startup: it uses `NANOUSD_BACKEND` if set, otherwise looks for the default backend (`libnanousd.so` / `.dylib`) next to the `nanousdapi` library. When you link `nanousdapi` **statically** into your own executable, set `NANOUSD_BACKEND` (or put the backend on your loader path) so it can be found — this applies to the default backend, not only the OpenUSD backend.

## Testing

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Debug -DNANOUSD_BUILD_TESTS=ON
cmake --build build --parallel
cd build && ctest --output-on-failure
```

There are two test suites:

- **`nanousd_tests`** — C++17 tests exercising parser internals, stage operations, and write support
- **`compliance_test`** — Pure C11 tests exercising only the public C API (`nanousdapi.h`)

Tests must be run from the repository root — both suites load compliance fixtures by repo-root-relative paths. `ctest` handles this automatically (CMake sets each test's `WORKING_DIRECTORY` to the repo root). **If you run a test binary directly** (e.g. `./build/compliance_test`) from another directory, it can't find the fixtures and many cases will appear to fail — `cd` to the repo root first. From the root, all cases pass.

### Sanitizer builds

AddressSanitizer, UndefinedBehaviorSanitizer, and ThreadSanitizer can be enabled for dev/CI test runs with GCC or Clang on Linux/macOS. AddressSanitizer and ThreadSanitizer are mutually exclusive; use `address;undefined` for ASan/UBSan or `thread` for TSan, not both together:

```bash
CC=clang CXX=clang++ cmake -B build-sanitize -DCMAKE_BUILD_TYPE=RelWithDebInfo \
    -DNANOUSD_BUILD_TESTS=ON \
    -DNANOUSD_ENABLE_SANITIZERS=ON
cmake --build build-sanitize --parallel
ctest --test-dir build-sanitize --output-on-failure
```

CMake configures CTest to run sanitizer tests with leak detection enabled and the repository's LeakSanitizer suppressions. Existing `ASAN_OPTIONS`, `LSAN_OPTIONS`, and `UBSAN_OPTIONS` from the configure environment are appended to those defaults. To run a test binary directly from the repository root, use the same environment:

```bash
ASAN_OPTIONS=halt_on_error=1:detect_leaks=1 \
LSAN_OPTIONS=suppressions=$PWD/cmake/sanitizers/lsan.supp:print_suppressions=0 \
UBSAN_OPTIONS=halt_on_error=1:print_stacktrace=1 \
./build-sanitize/RelWithDebInfo/compliance_test
```

ThreadSanitizer builds use `-DNANOUSD_SANITIZERS=thread`. GCC builds do not support combining `thread` and `undefined` because that sanitizer runtime combination reports false positives around `std::async`; CMake rejects `thread\;undefined` for GCC. Existing `TSAN_OPTIONS` from the configure environment are appended to CTest's default `TSAN_OPTIONS`.

Sanitized builds are not installable; the install rules are disabled when `NANOUSD_ENABLE_SANITIZERS=ON`.
